# Streaming QA Framework

A Python test automation framework that checks if a streaming endpoint is healthy, fast enough, and can handle multiple users at the same time.

I built this to understand how QA works for video delivery systems — the kind used for live events, OTT platforms, etc. The idea was to go beyond just checking if a URL returns 200, and actually test things that matter in production: latency thresholds, concurrent load, redirect behavior, content consistency.

---

## What it tests

**Health checks** — runs against the stream URL on every test run:
- Is the endpoint reachable at all?
- Does it return HTTP 200?
- Is the response time under the allowed threshold?
- Does it return actual content (non-empty body)?
- Is the content type correct for a stream?
- Are there any redirect loops?

**Consistency checks** — catches flaky behavior:
- Do 3 consecutive requests all succeed? (intermittent failures = unstable edge node)
- Is latency stable across 5 requests, or does it spike randomly?

**Load simulation** — fires 50 requests across 10 concurrent threads:
- Does success rate stay above 90%?
- Does average latency stay under 3 seconds?
- What's the throughput in requests per second?

## Setup

You need Python 3.10 or above.

```bash
git clone https://github.com/vijaykumardandu/streaming-qa-framework.git
cd streaming-qa-framework

python -m venv venv
venv\Scripts\activate        # Windows
source venv/bin/activate     # Mac/Linux

pip install -r requirements.txt
```

---

## How to run

```bash
# run everything and get an HTML report
python run_tests.py

# run only the health checks
pytest tests/test_stream_health.py -v

# run only the load simulation
pytest tests/test_load_simulation.py -v
```

The HTML report gets saved in the `reports/` folder with a timestamp in the filename.

---

## Pointing it at a different stream

Open `config/settings.py` and change these values:

```python
STREAM_URL = "https://your-stream-url-here"

CONCURRENT_USERS = 10       # how many parallel users to simulate
TOTAL_REQUESTS = 50         # total requests across all users

MAX_ALLOWED_LATENCY_MS = 3000      # fail if avg latency goes above this
MIN_SUCCESS_RATE_PERCENT = 90      # fail if success rate drops below this
```

That's all you need to change — the rest of the framework adapts automatically.

## Project structure

```
streaming_qa_framework/
├── config/
│   └── settings.py               ← stream URL and thresholds, change these
├── tests/
│   ├── test_stream_health.py     ← availability and consistency checks
│   └── test_load_simulation.py   ← concurrent user load test
├── utils/
│   └── helpers.py                ← shared functions for HTTP requests and stats
├── reports/                      ← HTML reports saved here after each run
├── run_tests.py                  ← entry point, runs everything
└── requirements.txt
```
